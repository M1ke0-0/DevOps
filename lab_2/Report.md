# Отчет по лабораторной работе №2: Основы работы с Git

**Цель работы:** Изучить базовые и команды системы контроля версий Git, научиться создавать локальные и удаленные репозитории, выполнять коммиты, клонировать репозитории, а также разрешать конфликты слияния.

## Ход работы:

### 1. Базовая настройка Git
Перед началом работы была выполнена глобальная настройка параметров пользователя (имя пользователя, email и название ветки по умолчанию).
```bash
git config --global user.name "Student"
git config --global user.email "student@example.com"
git config --global init.defaultBranch master
```

### 2. Создание локального репозитория (п. 4.2.1)
Создана рабочая директория `repo_new`, в которой был инициализирован новый пустой Git-репозиторий.
```bash
mkdir repo_new
cd repo_new
git init
```

### 3. Добавление файлов в индекс и создание коммита (п. 4.2.2)
В репозитории был создан файл `file.txt` и пустая директория `empty` с файлом `.gitkeep` (поскольку Git не отслеживает пустые директории). После проверки статуса (`git status`) изменения были добавлены в индекс и зафиксированы с сообщением "First commit".
```bash
# Создание файлов file.txt и empty/.gitkeep
git add .
git commit -m "First commit"
git status # -> nothing to commit, working tree clean
```

### 4. Работа с удаленным репозиторием (п. 4.2.3)
Сначала была предпринята попытка использовать обычный репозиторий в качестве удаленного, но затем был создан репозиторий `repo_master` с флагом `--bare`, который выступает в качестве центрального (удаленного) репозитория. Локальный репозиторий `repo_new` был связан с ним, после чего локальная ветка `master` была отправлена на сервер.
```bash
mkdir repo_master
cd repo_master
git init --bare

cd ../repo_new
git remote add origin "../repo_master"
git push --set-upstream origin master
```

### 5. Имитация работы другого пользователя (п. 4.2.4)
Для тестирования репозитория был создан временный репозиторий `repo_test`, в котором был сделан тестовый коммит (с файлом `rand.txt`) и успешно отправлен в `repo_master`. Затем временный репозиторий был удален. Таким образом была проверена корректность работы bare-репозитория на прием пушей от разных клиентов.

### 6. Клонирование репозитория (п. 4.2.5)
Для имитации совместной работы репозиторий `repo_master` был склонирован в новую директорию `repo_cloned`.
```bash
git clone "./repo_master" "repo_cloned"
cd repo_cloned
git status # -> Your branch is up to date with 'origin/master'.
```

### 7. Создание и разрешение конфликта слияния (п. 4.2.6)
Для искусственного создания конфликта слияния (merge conflict) один и тот же файл `file.txt` был изменен в двух разных локальных репозиториях:
1. В `repo_cloned` был записан текст "Text from repo_cloned repository", создан коммит "second" и отправлен на сервер (`git push`).
2. В `repo_new` файл `file.txt` был изменен на "Text from repo_new repository", создан коммит "other second". При выполнении `git push` операция была отклонена, так как сервер содержал изменения, которых нет локально.

Для получения изменений с сервера была выполнена команда `git pull`, что ожидаемо привело к конфликту слияния в файле `file.txt`.
```bash
Auto-merging file.txt
CONFLICT (content): Merge conflict in file.txt
Automatic merge failed; fix conflicts and then commit the result.
```

Конфликт был разрешен: оставлены изменения из обоих репозиториев (текст из `repo_new` на одной строке, а из `repo_cloned` — на другой). После разрешения конфликта был выполнен коммит слияния и отправка на сервер:
```bash
git add file.txt
git commit -m "Merged"
git push
```
История коммитов (`git log -n 4`) подтверждает успешное создание двух параллельных коммитов и успешный коммит слияния (Merge commit).
